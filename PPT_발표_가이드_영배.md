## 📊 4. 클라우드 인프라 소개 (2분)

### 🎤 발표 스크립트
```
"안녕하세요, 인프라를 담당한 영배입니다. 

저희가 왜 클라우드를 선택했는지부터 말씀드리겠습니다.
티켓팅 서비스는 특성상 평상시에는 트래픽이 적지만, 
티켓 오픈 시점에는 급격한 트래픽 증가가 발생합니다.

온프레미스 환경에서는 이런 트래픽 변화에 대응하기 어렵지만,
클라우드를 활용하면 필요할 때만 리소스를 확장할 수 있습니다."
```

### 📋 슬라이드 구성
1. **문제 정의**
   - 티켓팅 서비스의 트래픽 특성
   - 기존 온프레미스의 한계

2. **클라우드 선택 이유**
   - 탄력적 확장성
   - 비용 효율성
   - 고가용성

3. **아키텍처 개요**
   - 마이크로서비스 구조
   - AWS 주요 서비스 활용

### 🎨 시각 자료
- 트래픽 패턴 그래프 (평상시 vs 티켓오픈시)
- 전체 아키텍처 다이어그램
- AWS 서비스 아이콘들

## ⚙️ 5. EKS 구성 (3분)

### 🎤 발표 스크립트
```
"이제 구체적인 EKS 구성에 대해 설명드리겠습니다.

저희는 컨테이너 오케스트레이션을 위해 Kubernetes를 선택했고,
AWS의 관리형 서비스인 EKS를 활용했습니다.

특히 고가용성을 위해 Multi-AZ 구성을 적용했으며,
각 서비스를 독립적인 컨테이너로 분리하여 배포했습니다."
```

### 📋 슬라이드 구성
1. **왜 Kubernetes인가?**
   - 마이크로서비스 관리
   - 자동 복구 및 스케일링
   - 선언적 배포

2. **EKS 클러스터 구성**
   - Multi-AZ 배치 (ap-northeast-2a, 2b)
   - 워커 노드 구성
   - 네트워크 설정

3. **서비스별 배포 구조**
   - API, Gateway, Pay 서비스 분리
   - ConfigMap/Secret 활용
   - Service Mesh 구성

4. **Karpenter 도입**
   - 기존 Cluster Autoscaler 대비 장점
   - 빠른 노드 프로비저닝
   - 비용 최적화

### 🎨 시각 자료
- EKS 클러스터 아키텍처 다이어그램
- Multi-AZ 배치 구조도
- 서비스별 Pod 배치 현황
- Karpenter vs Cluster Autoscaler 비교표

## 📈 6. 오토스케일링 (2-3분)

### 🎤 발표 스크립트
```
"마지막으로 오토스케일링 구현에 대해 말씀드리겠습니다.

저희는 3단계 오토스케일링을 구현했습니다.
첫 번째는 HPA를 통한 Pod 단위 스케일링,
두 번째는 Karpenter를 통한 Node 단위 스케일링,
세 번째는 시간 기반 예측적 스케일링입니다.

특히 GitOps 환경에서 ArgoCD와 충돌하지 않도록
HPA의 minReplicas를 조정하는 방식을 채택했습니다."
```

### 📋 슬라이드 구성
1. **3단계 오토스케일링**
   - Pod Level: HPA (CPU/Memory 기반)
   - Node Level: Karpenter (자동 노드 프로비저닝)
   - Schedule Level: CronJob (시간 기반 예측)

2. **HPA 설정**
   - CPU 70%, Memory 80% 임계값
   - 각 서비스별 차별화된 설정
   - Scaling 정책 (스케일업/다운 속도)

3. **Karpenter 설정**
   - NodePool 구성 (default, peak-time)
   - Instance type 선택 전략
   - Consolidation 정책

4. **GitOps와의 통합**
   - ArgoCD selfHeal 문제 해결
   - CronJob을 통한 HPA 조작
   - 시간 기반 스케일링 시나리오

5. **성과 및 최적화**
   - 초기 문제: 과도한 노드 생성 (18개 파드 → 13개 노드)
   - 해결책: TopologySpreadConstraints 완화
   - 최종 결과: 효율적인 노드 활용

### 🎨 시각 자료
- 3단계 스케일링 플로우차트
- HPA 메트릭 대시보드 스크린샷
- 노드 생성/삭제 패턴 그래프
- Before/After 비교 (노드 사용량)

## 📚 예상 질문 및 답변 준비

### Q1: "왜 EKS를 선택했나요? ECS와 비교했을 때 장점은?"
**A**: "EKS를 선택한 이유는 첫째, Kubernetes의 표준화된 API로 벤더 종속성을 피할 수 있고, 둘째, 마이크로서비스 관리에 특화된 기능들(Service Discovery, ConfigMap 등)을 활용할 수 있기 때문입니다. ECS 대비 학습곡선은 있지만, 장기적으로 더 유연한 확장이 가능합니다."

### Q2: "Karpenter와 Cluster Autoscaler의 차이점은?"
**A**: "가장 큰 차이는 프로비저닝 속도입니다. Cluster Autoscaler는 기존 Auto Scaling Group을 확장하는 방식이라 2-3분이 걸리지만, Karpenter는 직접 EC2 API를 호출해서 30초 내에 노드를 생성할 수 있습니다. 또한 인스턴스 타입 선택도 더 유연합니다."

### Q3: "TopologySpreadConstraints 문제를 어떻게 해결했나요?"
**A**: "초기에는 엄격한 분산 정책으로 인해 노드가 계속 생성/삭제되는 문제가 있었습니다. maxSkew 값을 1에서 10으로 완화하고, whenUnsatisfiable을 ScheduleAnyway로 설정하여 평상시에는 효율성을, 스케일링 시에는 가용성을 확보했습니다."
